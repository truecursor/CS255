# CS255: DriverPass

## Project Summary

The DriverPass project was about designing a system for a company that helps students prepare for their driving tests. The client was DriverPass, and they wanted an online system that could handle student accounts, lesson scheduling, training packages, payments, practice tests, progress tracking, staff access, and DMV updates. The system also needed to support different users, including customers, secretaries, drivers, administrators, and outside services like the DMV system.

## What I Did Well

One thing I think I did well was breaking the project into clear requirements. Instead of only thinking about the system as a basic website, I looked at what each user needed to do. For example, customers needed to schedule lessons and take practice tests, while staff needed to manage appointments, packages, reports, and user access. I also think my diagrams helped organize the project because they showed how the users, system features, and stored information connect.

## What I Would Revise

If I could revise one part of my work, I would probably improve the diagrams in the system design document. I think the main ideas were there, but I could make some labels and relationships cleaner so they are easier to understand right away. I would also try to keep the diagrams detailed without making them too crowded, because a diagram should help explain the system instead of making it harder to follow.

## Interpreting User Needs

I interpreted the user's needs by looking at what DriverPass wanted the system to actually support. Students needed account access, package registration, lesson scheduling, practice tests, and progress tracking. Staff needed tools for scheduling, reports, driver notes, package management, and account access. I used those needs to guide the system design so the features matched the real tasks users would complete. Considering user needs is important because a system can look good on paper, but it will not be useful if it does not solve the client’s actual problems.

## My Approach to Designing Software

My approach to designing software is to understand the problem first before jumping into the design. I would start by reviewing the client's goals, identifying the main users, separating functional and nonfunctional requirements, and then using diagrams to organize the system. In the future, I would keep using tools like use case diagrams, activity diagrams, sequence diagrams, and class diagrams because they make the design easier to explain. I would also want more user feedback earlier in the process so I can catch unclear requirements before the system gets too far into development.
